# FPGA UART FIFO Project

A SystemVerilog UART loopback project with synchronous TX/RX FIFOs, a self-checking Verilator testbench, and a clean GitHub-ready structure. The project includes a UART transmitter, UART receiver, synchronous FIFO blocks, a FIFO-backed UART top module, and a testbench that verifies ordered byte loopback through the full datapath.[1][2]

## Project overview

This repository implements a UART datapath in SystemVerilog with:

- `uart_tx.sv`: UART transmitter.
- `uart_rx.sv`: UART receiver.
- `sync_fifo.sv`: synchronous FIFO used on TX and RX sides.
- `uart_fifo_top.sv`: top-level integration of UART TX, UART RX, TX FIFO, and RX FIFO.
- `tb_uart_fifo_top.sv`: self-checking testbench for loopback verification.[2][3]

The current verified flow pushes three bytes into the TX FIFO, transmits them through the UART path, loops the serial output back into the receiver, stores received bytes into the RX FIFO, and checks that the popped RX data matches the expected byte sequence exactly.[3][1]

## Features

- Parameterized UART clock frequency and baud rate.
- Parameterized FIFO depth.
- Separate TX and RX FIFOs around the UART datapath.
- Verilator-compatible self-checking testbench.
- VCD waveform dump for debug.
- Clean strict RX checking with no workaround path that masks regressions.[3][4]

## Design summary

### UART transmitter

The transmitter accepts `tx_start` and `tx_data`, serializes one start bit, eight data bits, and one stop bit, and exposes `busy` and `tx_done` status signals. The final working behavior depends on the transmitter only accepting `tx_start` while it is idle, which is why the top-level FIFO handoff must respect the `busy` transition timing.[6][7]

### UART receiver

The receiver synchronizes the incoming serial line, validates the start bit near the middle of the bit period, shifts in 8 data bits, and checks the stop bit. It outputs `rx_data`, `rx_valid`, and `framing_error`.[2][8]

### Synchronous FIFO

The FIFO uses synchronous read behavior, meaning `out_data` updates after a read enable event rather than combinationally in the same cycle. That read latency was central to the debug process and required the TX manager FSM to wait for valid FIFO output before asserting `tx_start`.[9][10]

### UART FIFO top

The top-level module manages:

- writes into the TX FIFO,
- handoff from TX FIFO to UART TX,
- serial loop handling through the UART blocks,
- writes from UART RX into the RX FIFO,
- reads from the RX FIFO through the testbench.[2][11]

The final stable TX manager behavior needs two protections:

1. Wait for synchronous FIFO output to become valid after `tx_fifo_rd_en`.
2. Wait for `tx_busy` to actually assert after `tx_start` before allowing the next FIFO pop.[10][12]

## Debugging history

The project originally showed multiple simulation issues before converging to the passing result.

### Issue 1: misleading TX monitor behavior

An earlier testbench revision launched multiple monitor tasks in parallel against the same serial line, causing multiple decoders to watch the same start bit and report confusing duplicated output. The testbench was simplified to rely on strict RX FIFO checking instead.[3][13]

### Issue 2: transmitter data-path bug

A transmitter-side issue caused all sampled bits to appear low and the stop bit check to fail. That was addressed before the final FIFO handoff debugging stage.[14][7]

### Issue 3: phantom `0x00` at RX

A phantom first byte appeared because the TX FSM captured FIFO output one cycle too early. Since the FIFO read was synchronous, `tx_fifo_dout` still held stale data when sampled, so the first transmitted byte was an old reset value instead of the intended FIFO entry.[9][10]

### Issue 4: skipped middle byte

After fixing the first latency issue, a second race appeared: `tx_start` was asserted and the FSM returned to idle before `tx_busy` became visible, allowing a premature extra FIFO pop. That caused a real queued byte to be removed from the FIFO while the UART transmitter was already busy, so the byte was silently lost.[12][6]

### Final fix

The final passing version waits for FIFO read latency and also waits for `tx_busy` to rise after `tx_start`. That produces a strict 1:1 relationship between FIFO pops and actual transmitted bytes.[10][6]

## Testbench behavior

The testbench:

- generates the clock,
- applies reset,
- pushes `0xA5`, `0x3C`, and `0xF0` into the TX FIFO,
- waits for RX FIFO data,
- pops RX FIFO data,
- checks exact byte equality in order,
- fails immediately on mismatch or timeout.[3][13]

The old `skip_initial_rx_00` workaround was intentionally removed after the RTL fix so that future regressions are not hidden by testbench exceptions.[15][16]

## Requirements

You need:

- Verilator installed.
- A C++ toolchain available to build the generated simulator.
- A shell environment such as Linux, WSL, or Git Bash-compatible tooling depending on the platform.[1][5]

## Build and run

### Lint

```bash
verilator --lint-only --timing -sv \
  --top-module tb_uart_fifo_top \
  rtl/sync_fifo.sv rtl/uart_tx.sv rtl/uart_rx.sv rtl/uart_fifo_top.sv tb/tb_uart_fifo_top.sv
```

### Build simulation

```bash
/usr/bin/verilator --trace --binary --timing -sv \
  --top-module tb_uart_fifo_top \
  rtl/sync_fifo.sv rtl/uart_tx.sv rtl/uart_rx.sv rtl/uart_fifo_top.sv tb/tb_uart_fifo_top.sv
```

### Run simulation

```bash
./obj_dir/Vtb_uart_fifo_top
```

### View waveform

The testbench writes `tb_uart_fifo_top.vcd`, which can be opened in a waveform viewer such as GTKWave after the run completes.[2]

## Troubleshooting

### Verilator `INITIALDLY` warning in the testbench

If Verilator reports `INITIALDLY` on task-driven assignments, use blocking assignments in the testbench task flow rather than non-blocking assignments inside `initial`-style procedural code.[4][21]

### `Clock skew detected` during build

That warning is usually caused by timestamp differences in the environment, especially on WSL or shared filesystems. If the simulation binary rebuilds correctly and the test passes, it is typically an environment timestamp issue rather than a design problem.[5][1]

### Push rejected on first GitHub upload

If GitHub rejects the push because the remote already contains files, either create a new empty repository or reconcile the remote history before pushing. The cleanest path for an existing project is usually an empty remote repository.[17][22]
